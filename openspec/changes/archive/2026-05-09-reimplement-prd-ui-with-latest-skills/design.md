## Context

项目已存在基于三份 PRD 与六张截图的前端实现，但当前输入资产命名存在“PRD 前缀与 UI 文件名前缀不一致”的问题，影响自动化配对稳定性。页面代码在整体结构与交互上已接近目标，但样式层仍有少量硬编码颜色，且未明确落地“本轮重实现”对应的 OpenSpec 变更证据。

本次设计目标是在不破坏现有页面信息架构的前提下，完成“输入资产规范化 + UI 细节对齐 + 规范化验收与归档”。

## Goals / Non-Goals

**Goals:**

- 建立严格可解析的 PRD/UI 同前缀配对关系，消除语义近似映射依赖。
- 保持三页核心交互不回退，并对齐截图的关键结构与视觉状态。
- 将主题相关样式统一收敛到变量体系，避免新增硬编码主题色。
- 输出可追溯的分析清单、问题清单、质量门禁结果与归档记录。

**Non-Goals:**

- 不引入新的业务页面或新业务流程。
- 不接入真实后端接口，继续使用 mock 数据策略。
- 不处理截图中未提供的全状态（如完整 hover/disabled 细节）的精细视觉扩展。

## Decisions

1. 输入资产命名采用“修正 UI 文件名”的方式达成同前缀配对。
- 原因：比修改 PRD 内容更稳健，且能直接满足自动匹配策略。
- 备选方案：修改 PRD 资产字段映射到旧文件名。未选原因：会保留历史命名偏差，不利于后续自动化批处理。

2. 页面实现采用“最小必要改动”策略。
- 原因：现有结构已覆盖主要流程，重实现目标是规范化和一致性，不是推翻重写。
- 备选方案：全量重构为新组件体系。未选原因：改动过大、风险高，超出本次需求。

3. 样式统一使用变量体系，主题主色通过 Antd token 或变量保证一致。
- 原因：满足样式规范，并降低后续主题切换与统一改色成本。
- 备选方案：仅替换个别硬编码色值。未选原因：仍会残留风格漂移风险。

4. 验收输出延续“分析清单 + 问题清单”双文档。
- 原因：与既有流程一致，且便于后续审计与回归。
- 备选方案：仅保留问题清单。未选原因：缺少分析基线，不利于追溯。

## Risks / Trade-offs

- [风险] 截图仅覆盖默认态，交互细节可能存在未观察差异。→ 缓解：在问题清单标记 `待补状态截图`，将其作为非阻断待办。
- [风险] 修改文件命名后，文档引用可能遗漏。→ 缓解：批量检索 `docs/ui/` 旧文件名并统一替换。
- [风险] 样式变量替换可能引发局部视觉变化。→ 缓解：仅替换主题相关硬编码色值，保留布局参数不做无关调整。
