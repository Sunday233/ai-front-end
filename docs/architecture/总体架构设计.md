# PRD + UI 自动实现总体架构设计

## 1. 文档目标

本文档定义本仓库的目标形态：当用户输入“帮我实现 docs 文件下的 PRD 和 UI 设计稿截图”或同义语句时，Agent 能够完成从意图识别、方案规划、OpenSpec 变更、代码实现到上线就绪校验的端到端自动执行。

## 2. 现状分析（基于全仓库扫描）

### 2.1 当前仓库定位

当前仓库是“规则 + 技能 + OpenSpec”治理仓库，不是可直接运行的业务前端工程。

- 存在规则体系：`.github/instructions/*.instructions.md`
- 存在技能体系：`.github/skills/*/SKILL.md`
- 存在 OpenSpec 流程：`openspec/AGENTS.md`、`openspec/project.md`
- 不存在业务工程代码：无 `src/`、无 `package.json`、无 `vite.config.*`

### 2.2 当前可复用能力

1. 规范治理能力：对组件、路由、API、状态、样式、测试有明确规则。
2. 提案流程能力：具备 OpenSpec 三阶段流程与验证约束。
3. 设计相关能力：已有设计分析与 UI 验收技能。

### 2.3 当前缺口

1. 缺少“PRD + UI -> 自动落地”的统一入口规范。
2. 缺少针对 `docs/prd` 与 `docs/ui` 的输入协议与命名协议。
3. 缺少“多 PRD 批处理、失败续跑、重试、高风险确认”的统一执行策略。
4. 缺少标准化执行日志模板与上线就绪门禁描述。

## 3. 目标架构（To-Be）

## 3.1 架构分层

1. 输入资产层
- `docs/prd/`：PRD 文档输入
- `docs/ui/`：UI 截图输入
- 命名约束：按功能 `slug` 同名前缀配对

2. 意图识别层
- 识别规则：语义识别，不依赖固定句式
- 触发条件：同时满足“PRD 输入 + UI 输入 + 实现意图”

3. 计划治理层（OpenSpec）
- 强制先进入 Proposal 阶段：创建/更新 OpenSpec 变更（`proposal.md`、`tasks.md`、`spec delta`）
- Proposal 阶段通过严格校验后，默认确认进入 Apply 阶段
- Apply 阶段按同一份 `proposal/tasks/spec` 实施，不允许脱离 spec
- 验收通过后进入 Archive 阶段并归档，沉淀后续上下文

4. 实现编排层（Skills + Rules）
- design-analysis：先抽取 UI 分析清单
- create-proposal：形成执行任务与规格边界
- create-route/create-component/create-api/create-store/theme-variables：按规范落地
- ui-verification：完成 UI 还原验收

5. 质量门禁层（上线就绪）
- TypeScript：零错误
- ESLint：通过
- 测试：关键路径单测 + 关键页面冒烟 + E2E
- UI：截图对比通过（截图优先，分析清单作为辅助）
- 构建：`vite build` 成功
- 可观测性：任务 ID、关键节点日志、错误分类与重试记录

6. 输出交付层
- 实现结果代码
- OpenSpec 变更资产
- UI 问题清单
- 执行日志

## 3.2 关键执行策略

### 3.2.1 多 PRD 执行策略

1. 按文件名排序逐个执行 PRD。
2. 每个 PRD 按同名前缀匹配 `docs/ui` 截图集合。
3. UI 缺失时继续实现，并在风险清单中标记“UI 待补”。

### 3.2.2 失败与重试策略

1. 编译/测试失败自动修复，最多重试 2 次。
2. 单 PRD 失败不阻断后续 PRD，最终汇总失败项。

### 3.2.3 OpenSpec 生命周期策略

1. Proposal：先执行 `openspec proposal`（逻辑命令）创建提案；CLI 无该命令时，等价执行 `openspec new change <change-id>` + 生成 `proposal/tasks/spec` + `openspec validate --strict`。
2. Apply：Proposal 通过后默认确认进入 `openspec apply`（逻辑命令）；CLI 无该命令时，等价执行为按 `tasks.md` 逐项实施并回写任务状态。
3. Archive：验收通过后执行 `openspec archive <change-id> --yes`，并通过 `openspec validate --strict` 确保归档后状态可追溯。

### 3.2.4 高风险确认策略

以下操作必须人工确认：

1. 大范围重构（跨多个模块）。
2. 删除或重命名核心文件。
3. 修改全局规范（instructions/skills）。
4. 主版本升级或核心依赖变更。

## 3.3 上线就绪定义

本轮目标是“上线就绪”，不要求立即部署。上线就绪满足：

1. 构建产物可生成且可在标准静态托管环境部署。
2. 必要质量门禁全部通过。
3. 可观测性与兜底策略完整。
4. 关键风险可追踪且有缓解方案。

## 5. 角色职责划分

1. 用户
- 提供 PRD 与 UI 资产
- 对高风险操作做确认

2. Agent
- 自动解析输入资产
- 自动执行 OpenSpec + 实现 + 校验全流程
- 输出可审计日志与问题清单

3. 规则层（instructions）
- 约束技术栈、目录、命名、样式、测试

4. 技能层（skills）
- 提供分阶段可执行方法（分析、提案、实现、验收）

5. OpenSpec
- 作为变更治理与追踪主线

## 6. 本次改造范围

本次仅覆盖“文档与规则体系改造”，不包含业务页面/业务功能代码实现。
