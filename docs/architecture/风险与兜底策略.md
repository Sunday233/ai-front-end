# PRD + UI 自动实现风险与兜底策略

## 1. 风险分级

## 1.1 阻断级风险（必须拦截）

1. OpenSpec 校验持续失败，无法形成有效变更提案。
2. TypeScript/Lint/构建无法通过，重试 2 次后仍失败。
3. 核心规范被破坏（技术栈、目录结构、命名规则）。
4. 高风险操作未获确认即尝试执行。

## 1.2 非阻断风险（可留档继续）

1. UI 存在少量非关键差异（如文案微调、像素级细节）。
2. 某 PRD 缺失配对 UI 截图。
3. 后端接口未就绪，采用 mock 实现。

## 2. 主要风险场景与策略

## 2.1 PRD 与 UI 冲突

策略：UI 优先，PRD 作为业务语义参考。

执行规则：

1. 先按 UI 结构实现页面骨架与交互层级。
2. PRD 中与 UI 冲突项记录到“冲突清单”。
3. 在结果汇总中显式标注待产品确认项。

## 2.2 UI 资产缺失或配对失败

策略：继续实现，不阻断主流程。

执行规则：

1. 若无同名前缀截图，按 PRD 先实现结构与交互。
2. 在日志中记录 `UI_PENDING` 标记。
3. 在待办中自动生成“补图后复验”任务。

## 2.3 自动修复失败

策略：限制重试，避免死循环。

执行规则：

1. 每个 PRD 质量修复最多重试 2 次。
2. 重试耗尽后标记 `FAILED_WITH_RETRY_EXHAUSTED`。
3. 继续处理后续 PRD，最终统一汇总失败项。

## 2.4 规范污染风险

策略：对治理文件改动设置高风险确认。

高风险触发：

1. 修改 `.github/instructions`。
2. 修改 `.github/skills`。
3. 删除/重命名核心规范文件。
4. 主版本升级或核心依赖变更。

## 3. 兜底机制

## 3.1 技术兜底

1. 强制走 OpenSpec 严格校验。
2. 强制走类型、lint、测试、构建门禁。
3. 强制产出执行日志与风险清单。

## 3.2 业务兜底

1. 缺少 UI 时先交付可运行结构。
2. 接口未就绪时使用 mock，附替换清单。
3. 保留非阻断问题，避免卡死迭代。

## 3.3 操作兜底

1. 高风险动作必须人工确认。
2. 单 PRD 失败不影响批次剩余 PRD。
3. 所有失败都必须可追踪到任务 ID。

## 4. 通过标准

满足以下条件可判定“上线就绪（非部署）”：

1. 阻断级风险全部关闭。
2. 质量门禁通过。
3. 非阻断风险已记录并有后续处理计划。
4. 输出材料完整（变更、清单、日志、结论）。
